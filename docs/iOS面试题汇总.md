## 1. 组件封装：最得意的组件/二次封装有哪些？特点是什么？

### 1.1 常见可封装的基础能力（示例）

- **文件与存储**：文件读写、目录管理、文件存在性判断、缓存清理策略。
- **权限管理**：相机/相册/定位/通知/蓝牙等权限状态读取、统一弹窗引导、降级方案。
- **网络层**：HTTP 封装、统一 header、签名、重试/超时、错误码映射、日志与抓包开关。
- **加密与安全**：AES/DES、请求签名、敏感数据脱敏、Keychain/安全存储封装。
- **JSON 解析**：统一模型层、容错字段、版本兼容。
- **定位/地图/蓝牙**：多 SDK 适配、回调封装、统一状态机。
- **动画/绘图**：贝塞尔曲线、复杂动效组件化、可配置参数。

### 1.2 突出特点怎么讲（面试口径）

- **复用**：跨项目可直接搬运，减少重复造轮子。
- **一致性**：统一接口/错误处理/日志埋点，减少“每个人写一套”的分裂。
- **更轻更可控**：相对第三方库更贴合业务，API 更简洁，升级风险更可控。
- **效率**：节省写代码、调试、测试时间；核心能力沉淀后迭代更快。

## 2. 陌生模块 bug 怎么排查？（步骤写出来）

### 2.1 建议步骤

1. **复现问题**：明确触发路径、输入数据、机型/系统版本/网络环境。
2. **定位范围**：从崩溃堆栈/报错日志入手；若无报错，先缩小到最小可疑模块。
3. **断点 + 日志**：关键路径逐步走，打印关键参数与状态（类型、空值、边界值）。
4. **排除法**：必要时注释/开关二分，快速确认问题发生在哪个代码块。
5. **修复与验证**：补充边界处理，回归原路径与相关路径。
6. **记录沉淀**：写到复盘/知识库，避免同类问题反复出现。

### 2.2 面试加分点

- **“先复现再修”**：不凭猜测改代码。
- **“先缩小范围再深入”**：减少盲目全局搜索/乱改。
- **“修复要带验证”**：回归 + 相关路径 + 线上防护（兜底/监控）。

## 3. 团队评优：你能接受的方式？别人不服气的原因？你不服气怎么处理？

## 4. 系统设计题：做一个类腾讯视频播放器（含投屏），怎么拆任务与排期？先做什么？

### 4.1 任务拆解

- **页面设计**：播放器壳、控制层、列表/详情、投屏控制页。
- **播放功能**：播放/暂停/快进/倍速/清晰度/全屏、断点续播、错误重试。
- **投屏功能**：设备发现、连接、投屏协议、播放控制同步、异常处理（断连/切网）。
- **测试与调优**：弱网、后台/前台切换、耗电、崩溃、兼容性。

### 4.2 排期示例（可按项目调整）

1. **第一阶段：本地播放链路**（约 1 周）：播放器内核接入 + 基本交互 + 播放稳定性。
2. **第二阶段：投屏流程**（约 1 周）：发现/连接 + 投屏控制页面 + 基本状态同步。
3. **第三阶段：联调与工程收口**（约 2 天）：接口、跳转、权限、账号态等衔接。
4. **第四阶段：测试与修复**（约 1 周）：兼容性/弱网/异常场景覆盖。
5. **后续迭代**：评论/分享/下载/弹幕等增值功能分期。

### 4.3 进入开发阶段先做什么

- **先打通“最短可用链路（MVP）”**：能播 → 能控 → 能稳定；并确定播放器内核与投屏协议/依赖。

## 5. 手写题：随机数量的 50×50 方块放入 300×600，互不重叠（任意语言）

### 5.1 思路

- 随机生成坐标 \((x,y)\)，构造矩形区域。
- 与已存在矩形做 **碰撞检测**（重叠就重试）。
- 设定重试上限，避免极端情况下死循环。

> 备注：你原始答案用 Flutter 给了示例实现（用 `Rect.overlaps`），思路对 OC/Swift 也同样适用。

## 6. IM 群消息：撤回/删除 + 本地缓存，如何保证多端一致性？

### 6.1 状态字段与事件驱动

- 每条消息维护独立状态字段：**deleted / recalled / read / delivered** 等。
- 服务端以“事件/通知”形式推送状态变更（透传/长连），客户端收到后更新本地缓存与 UI。

### 6.2 本地缓存与增量对齐

- 聊天记录本地落库。
- 用 **服务端 maxId** 与本地最新 maxId 对比，拉取 **增量更新**，保证断网/多端后能对齐。

## 7. 组件设计题：图片预览组件（含下载与缓存），怎么做？注意什么？

### 7.1 实现思路

- 先评估成熟方案（例如 iOS 常见的图片浏览库、或从历史项目复用），再按业务做 UI/交互微调。
- 下载与缓存用成熟框架（iOS 常用 SDWebImage 一类），并补齐缓存策略（内存/磁盘、过期、清理）。

### 7.2 关键注意事项

- **通用性**：后续可能扩展到文件预览（PDF/视频/Office）。
- **转场动画**：微信式“从缩略图放大”的过渡需要计算起始/结束 Rect，交互细节很考验体验。
- **异常与性能**：大图、长图、弱网占位、OOM 风险、预加载与手势冲突。

## 8. 上传下载任务列表：前端轮询 vs 服务端长连接推送，选哪个？

### 8.1 结论

- 更倾向 **长连接推送更新**。

### 8.2 原因

- 任务状态/进度/速率是**高频变化**，轮询会带来：
  - 请求频繁、服务器压力大
  - 客户端耗电/网络开销大
- 长连推送：
  - 客户端监听即可
  - 服务端按 taskId 精准推送变更，客户端可按 id 局部刷新

### 9.1 用 `chain-ios` 回答“工程能力/架构与模块化”

可以重点围绕这些面试点来讲：

- **模块化/分层**：业务模块、基础能力模块（network/storage/logging）如何拆；模块边界怎么定；如何避免循环依赖。
- **依赖管理（CocoaPods）**：`Podfile`、`Podfile.lock` 的使用习惯；如何处理三方库升级、版本冲突、二进制化/私有 Pod。
- **环境与构建产物**：`Debug/Release` 配置、构建脚本、打包与签名（能讲“可复制的流水线”更加分）。
- **稳定性兜底**：崩溃收集、日志分级、关键链路埋点、灰度与回滚思路。

### 9.2 `chain-ios` 项目优势与不足（可背口径）

#### 优势（怎么夸，夸到点上）

- **分层意识清晰**：目录里能看到 `Controllers / Views / ViewModels / Models` 的组织方式（例如 SpotDetail 下的 Controller/View/ViewModel），便于多人协作与定位问题。
- **基础能力覆盖完整**：`AppDelegate` 启动链路里有网络统一配置、Toast/HUD、主题、键盘管理、推送、播放器/WebView 池等（能说明“工程体系”比较齐）。
- **网络与全局 Header 管理**：有统一的 `NetWorkTool.header(...)` 与网络监测/配置更新逻辑，面试时可讲“请求签名/鉴权 header/缓存策略怎么落地”。
- **多语言与主题**：存在 `Localizable.strings` 多语言资源与 `SwiftTheme`，可讲国际化与暗黑模式工程化。
- **依赖成熟**：Podfile 里覆盖网络、布局、图片缓存、骨架屏、富文本、图表、数据库（WCDB）等常见能力，能体现落地经验。

#### 不足（说风险 + 给改进方向，面试更稳）

- **启动与全局初始化偏重**：`AppDelegate` 承担了大量初始化与业务订阅（通知监听、登录态切换等），容易变成“巨石入口”，建议拆成 `AppBootstrapper/Service` 按域初始化并做懒加载。
- **依赖管理偏重且耦合 CocoaPods**：Pod 数量多、`use_frameworks!` + `inhibit_all_warnings!` 会掩盖告警；可补充“逐步收敛依赖、关键库用 SPM/二进制化、告警分级治理”的策略。
- **全局单例/通知驱动较多**：看到 `SystemManager/UserManager/...` 一类全局状态与 Notification/Rx 订阅，规模变大后会加重耦合；可引入更明确的依赖注入与模块边界。
- **测试与质量体系不明显**：可补充“核心业务单测、网络层契约测试、CI 检查（lint/format）”作为改进计划。

### 9.3 用 `lldb_commands` 回答“调试提效（性能/稳定性/疑难排查）”

面试时不要背命令本身，讲“方法论 + 工具组合”更稳：

- **卡顿/掉帧**：
  - 先定性：是主线程阻塞、还是渲染/布局开销、还是 IO/锁竞争。
  - 工具链：`Instruments(Time Profiler/Core Animation)` + 关键函数断点 + LLDB 辅助定位热点。
- **内存问题（OOM/泄漏）**：
  - 工具链：`Instruments(Leaks/Allocations)`；必要时用 LLDB 辅助检查对象图与引用链。
- **线上问题快速定位**：
  - 先复现路径与最小用例；再用符号化崩溃栈、日志关键字、断点与条件断点缩小范围。
  - LLDB 脚本的价值：把常用排查动作“脚本化/标准化”（比如常用查询/格式化输出/快速定位对象信息），提升定位效率并可团队复用。

### 9.4 面试时的“可复用一句话”

> 我准备面试题不会只背概念，而是会结合工程来讲：在 `chain-ios` 这种真实项目里我怎么做模块边界、依赖管理与交付；遇到卡顿/内存/崩溃会用 Instruments 定位，再用 LLDB（脚本化命令）把定位效率提高并沉淀成团队可复用的工具链。
